# Upload Data Model

Status: additive foundation (phase 1).

## Lifecycle semantics

- Upload is not publish.
- Upload lifecycle status is separate from moderation and processing state.
- Public visibility is strict and requires both:
  - `status === "published"`
  - `visibility === "public"`
- This is the canonical definition of whether an asset is public.
- Moderation approval and processing readiness are workflow eligibility concerns, not part of canonical public visibility.

Lifecycle statuses in this additive model:

- `uploaded`
- `pending_review`
- `published`
- `rejected`
- `removed`

Visibility states in this additive model:

- `private`
- `public`

## User upload profile

The upload profile models upload eligibility and risk controls independently from the existing runtime user contract.

It includes:

- trust tier (`basic | trusted | restricted`)
- hard limits (daily, weekly, monthly, burst-per-hour)
- usage counters and window starts
- moderation health and upload suspension timestamp

Default tier profiles are encoded in `src/services/uploadDefaults.ts`.

## Upload asset document

The additive upload asset model is defined in `src/types/upload.ts` and includes:

- ownership and media metadata
- lifecycle status + visibility
- moderation state (`scanState`, `scanResult`, `finalDecision`)
- processing state (`metadataState`, `derivativeState`, `hashState`)
- metrics and timestamps

Initial defaults are generated by `createInitialAssetDoc` and always start private with uploaded lifecycle status.

Workflow helpers may enforce stricter publish eligibility (for example moderation approval), but that logic remains separate from canonical public visibility checks.

## Upload asset events

Asset event contracts are also defined in `src/types/upload.ts`.

Events provide an audit trail for transition and operator visibility, for example:

- upload completed
- submitted for review
- approved/rejected
- published/removed
- processing failed

`createAssetEvent` creates event docs with numeric timestamps.

## Storage path conventions

Helpers in `src/utils/uploadStoragePaths.ts` generate exact deterministic paths:

- `uploads/quarantine/{assetId}/original`
- `uploads/quarantine/{assetId}/preview`
- `uploads/quarantine/{assetId}/thumbnail`
- `uploads/public/{assetId}/original`
- `uploads/public/{assetId}/preview`
- `uploads/public/{assetId}/thumbnail`

## Compatibility note

Legacy meme statuses (`pending | approved | rejected`) remain operational in this phase.

Documentation-only mapping for future migration planning:

- `pending -> pending_review`
- `approved -> published`
- `rejected -> rejected`

This mapping is not runtime logic in this additive phase.
